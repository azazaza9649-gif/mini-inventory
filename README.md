# Mini Inventory API

Простий REST API для керування категоріями та предметами (мінімальний облік запасів).
Проєкт реалізується покроково в межах лабораторних робіт №0–№6 з прозорою історією комітів.

## Команда
- Індивідуальний проєкт (за потреби можна розширити до командного)

## Проблема
Потрібно вести невеликий структурований список категорій і предметів та виконувати базові операції створення, перегляду й видалення через HTTP-ендпоїнти.

## Основні сутності
- **Категорія (Category)**
  - `id: string`
  - `name: string`
  - `createdAt: datetime`
- **Предмет (Item)**
  - `id: string`
  - `name: string`
  - `quantity: number`
  - `categoryId: string` (зв’язок з Category)
  - `createdAt: datetime`

## API (заплановано)
### Категорії
- `POST /categories` — створити категорію
- `GET /categories` — отримати список категорій
- `DELETE /categories/:id` — видалити категорію (заборонено, якщо в категорії є предмети)

### Предмети
- `POST /items` — створити предмет
- `GET /items` — отримати список предметів (необов’язковий фільтр `?categoryId=...`)
- `DELETE /items/:id` — видалити предмет

### Health
- `GET /health` — перевірка доступності сервісу

## Технології (заплановано)
- TypeScript
- Node.js
- Fastify (HTTP)
- Prisma + база даних (SQLite локально, Postgres для деплою) — наступні лабораторні
- Тести: unit / integration / e2e + мутаційне тестування — наступні лабораторні
- CI/CD: GitHub Actions + деплой — наступні лабораторні

## План лабораторних робіт
- **ЛР0**: вибір ідеї, створення репозиторію, опис ідеї в README
- **ЛР1**: налаштування проєкту (пакети, стиль коду, форматування, лінтер, Git hooks)
- **ЛР2**: архітектура та діаграми (компоненти, ER, ключові сценарії)
- **ЛР3**: інтерактивний прототип зі статичними/in-memory даними (без БД)
- **ЛР4**: інтеграція з БД та заміна статичних даних на реальні
- **ЛР5**: unit + integration + e2e + мутаційне тестування та короткий звіт
- **ЛР6**: CI + CD та публічне розгортання (сервіс доступний з інтернету)
